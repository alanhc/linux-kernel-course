## 單元 0｜本週地圖

**導師：** 歡迎來到第一週。這週我們先不急著讀原始碼，而是先回答三個問題——為什麼要讀核心？我要具備什麼基礎？我怎麼把環境架起來、開始動手量測？把心態與工具準備好，後面十幾週才不會半途卡死。

**阿修：** 我以為第一週就要開始 trace `schedule()` 了。

**導師：** 那是很多人陣亡的地方。核心不是「背 API」，而是理解**取捨（trade-off）**。這週的目標，是讓你養成「凡事量測、凡事問為什麼」的習慣。

## 單元 1｜為什麼要讀 Linux 核心？

**阿修：** 老實說，我平常寫應用程式也活得好好的，為什麼要碰核心？

**導師：** 因為你「以為」你懂的事情，其實都是核心幫你撐住的。你呼叫 `malloc`，背後是誰跟實體記憶體要頁面？你 `printf`，字元怎麼跨越 user space 到硬體？你開兩個執行緒，誰決定哪個先跑？

**阿修：** ……我一直當作理所當然。

**導師：** 讀核心有三個具體收穫。第一，**除錯功力**：當你的服務在高負載下延遲暴衝，你能往下追到排程、鎖競爭、cache miss，而不是只會重開機。第二，**設計品味**：核心是幾十年、上億行使用者驗證過的工程取捨範本。第三，**可貢獻性**：它完全開源，你讀得懂就能改、能送 patch。

**阿修：** 所以重點不是「記住核心怎麼寫」，是「學會它怎麼想」。

**導師：** 正是。而且要動手——讀而不做，等於沒讀。

> 🎬 **視覺提示**〔id: syscall-boundary〕畫一條 user space / kernel space 分隔線，`malloc`、`printf`、`pthread_create` 的箭頭穿越到核心對應子系統。

### 🧠 自我評量 1
研讀 Linux 核心原始碼，最核心的學習目的是什麼？

- a) 背下常用核心函式的名稱與參數
- b) 理解系統在各種限制下所做的「工程取捨」，並能據此除錯與設計
- c) 記住每個檔案在原始碼樹的路徑

<details><summary>參考解答</summary>

**b**。核心的價值在於「為什麼這樣設計」——延遲與吞吐、記憶體與速度、公平與效率之間的取捨。a、c 是查得到的死知識，可靠工具檢索；真正稀缺的是取捨的判斷力。
</details>

## 單元 2｜先補齊 C 與底層基礎：未定義行為（UB）

**導師：** 核心幾乎全用 C 寫。而 C 最危險的不是語法，是**未定義行為（undefined behavior）**。你猜這段在做什麼？

```c
int a = INT_MAX;
a = a + 1;          // 有號整數溢位
```

**阿修：** 就……變成負的最小值？回繞（wrap around）嘛。

**導師：** 那是你「假設」硬體用二補數。但在 C 標準裡，**有號整數溢位是未定義行為**——編譯器有權假設它「永不發生」，於是把你的溢位檢查整段最佳化掉。這類 bug 在核心裡可能就是一個可被利用的資安漏洞。

**阿修：** 所以不能靠「我覺得它會 wrap」。

**導師：** 對。核心開發者的紀律是：**不依賴未定義行為**，需要回繞時用 `unsigned`（無號溢位才有明確定義），並清楚知道自己在做什麼。這週你要建立的直覺是——「這行程式碼的行為，是標準保證的，還是我運氣好？」

> 🎬 **視覺提示**〔id: signed-overflow〕數線動畫：`INT_MAX` 加一後，先顯示「你以為 → 最小值」，再打上紅框「標準 → 未定義」，並示意編譯器刪除檢查。

### 🧠 自我評量 2
關於 C 語言中 `INT_MAX + 1`，下列敘述何者正確？

- a) 標準保證回繞到 `INT_MIN`，可安全用來做溢位檢查
- b) 是未定義行為，編譯器可能假設它不發生並移除相關檢查
- c) 一定會讓程式當機

<details><summary>參考解答</summary>

**b**。有號整數溢位為未定義行為。若需要有定義的回繞語意，應改用無號型別（如 `unsigned int`），其溢位以 2ⁿ 取模是標準明訂的。c 不對——UB 不保證當機，反而常「看似正常」直到某次最佳化或平台改變才爆。
</details>

## 單元 3｜開發與量測環境：工具鏈

**阿修：** 環境我要裝什麼？

**導師：** 一組最小但夠用的工具鏈，全部開源：

- **gcc / clang**：編譯器。試著開 `-Wall -Wextra`，把警告當錯誤看待。
- **gdb**：除錯器。學會下中斷點、看 backtrace、印變數，比 `printf` 除錯強太多。
- **QEMU**：把核心跑在虛擬機裡，改壞了也不會弄死你的主機。
- **git**：不只版本控制，`git bisect` 能在上千次 commit 中二分搜出「哪一版把功能搞壞」。
- **make / perf**：建置與效能量測。

**阿修：** 為什麼特別強調量測？

**導師：** 因為核心的世界裡，**直覺經常是錯的**。你以為某個改動更快，實測可能更慢——因為 cache、分支預測、記憶體對齊在背後作祟。這門課的鐵律是：**先量測，再下結論**。

> 🎬 **視覺提示**〔id: toolchain-pipeline〕橫向流程圖：`原始碼 → gcc → QEMU 執行 → gdb 觀察 → perf 量測`，資料往右流動並在 `perf` 節點跳出一張延遲直方圖。

### 🧠 自我評量 3
你懷疑某次 commit 讓核心某功能出錯，但不確定是哪一次改動。最有效率的定位工具是？

- a) `git bisect`：以二分搜尋在提交歷史中快速定位出問題的 commit
- b) 從第一個 commit 逐一手動檢查
- c) 直接重寫整個模組

<details><summary>參考解答</summary>

**a**。`git bisect` 以 O(log n) 次測試定位問題 commit，n 次提交只需約 log₂(n) 次驗證。b 是 O(n) 的蠻力；c 沒有先找出根因，往往連原本哪裡對都不知道。
</details>

## 單元 4｜核心的巨觀結構速覽

**導師：** 最後，給你一張全局地圖，免得後面見樹不見林。Linux 核心大致分成五塊：

1. **行程管理與排程**：決定哪個任務用 CPU、用多久。
2. **記憶體管理**：虛擬記憶體、分頁、`kmalloc` 等配置器。
3. **檔案系統與 VFS**：用統一介面抽象各種儲存。
4. **網路協定堆疊**：從 socket 到網卡驅動。
5. **裝置驅動程式**：核心與硬體對話的邊界。

**阿修：** 這些之間怎麼互動？

**導師：** 靠**抽象層與明確介面**。例如 VFS 讓「開檔」對 ext4、Btrfs、網路檔案系統長得一樣；驅動模型讓核心不必知道你插的是哪牌硬碟。這種「用介面隔離變化」的手法，你在整門課會反覆看到——也是很好的軟體工程範本。

**阿修：** 那我們哪一塊會先深入？

**導師：** 從你最有感的**行程與排程**開始，第二週見。這也是為什麼我第一週要你先搞懂「取捨」——排程器整個就是公平與效率的拉鋸。

> 🎬 **視覺提示**〔id: kernel-subsystems〕心智圖：中心節點「Linux Kernel」向外連五個子系統，滑鼠移到「排程」時該節點放大並標記「Week 2 →」。

### 🧠 自我評量 4
VFS（虛擬檔案系統）在核心中扮演的角色，最貼切的描述是？

- a) 一種特定的磁碟檔案系統格式
- b) 一層抽象介面，讓上層以統一方式操作 ext4、Btrfs、網路檔案系統等不同實作
- c) 負責 CPU 排程的模組

<details><summary>參考解答</summary>

**b**。VFS 以統一的 `open/read/write` 等介面隔離底層差異，是「以介面隔離變化」的經典範例。a 錯把抽象層當成具體格式；c 是排程子系統的職責。
</details>

## 本週實作任務

1. **架環境**：安裝 gcc、gdb、QEMU、git，成功編譯並在 QEMU 啟動一個最小核心（或跑通課程指定的 hello 模組）。
2. **UB 實驗**：寫一支含有號整數溢位的小程式，分別以 `-O0` 和 `-O2` 編譯，觀察行為差異，寫下你的解讀。
3. **量測練習**：用 `perf stat` 量測任一小程式，記錄 cache-misses 與 instructions，貼上你的觀察。
4. **提問**：把本週四題自我評量中你最不確定的一題，改寫成你自己的版本，並附解答——之後可作為給同儕的題庫貢獻。
